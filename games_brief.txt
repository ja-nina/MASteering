This is the real crux, and you're right to force the check — "has a reusable implementation" and "has a language interface" are two different questions, and for most of these repos the answer diverges. Let me verify the two things that most affect the answer: whether the classic-games library exposes text, and whether there's an existing framework that already gives LLM/text interfaces to these games.You've identified the actual crux, and it's worth being precise: the unifying abstraction for an LLM testbed isn't the env interface (PettingZoo) — it's the language interface, i.e. a TextRenderer (state ? prompt) plus an ActionParser (LLM text ? valid env action, with error feedback on invalid output). Every game needs both. So the right question for each repo is "does it ship a language interface, or just an RL one?" — and I checked. The answer splits cleanly into three tiers.
Tier 1 — already language-native (reuse as-is)
These were built for LLM agents; text in/out is the point.
    • GovSim — NL negotiation + extraction decisions, agents are LLMs. 
    • SocialDeductionLLM (Among Us) — free-form NL discussion + voting. 
    • Welfare Diplomacy / diplobench — zero-shot prompted LLMs, NL negotiation, JSON orders. 
    • Werewolf/Avalon LLM arenas. 
Tier 2 — RL/symbolic, but the language interface is cheap or already wrapped
    • OpenSpiel games (PD, stag hunt, matrix/coordination): OpenSpiel already exposes text hooks — a string representing the observation for a player via observation_string() / information_state_string(), plus action_to_string() and legal_actions(). And there's an off-the-shelf wrapper: Game Reasoning Arena wraps the OpenSpiel game catalogue in a Gymnasium-like interface where, at each step, the OpenSpiel API's output is wrapped in a text prompt passed to the LLM agent as the state observation, including the current state representation, the legal actions, and for some games a summary of past moves. So for the classic games you get the language layer nearly for free. 
    • Your native symbolic games (GBS, beauty contest, minority, market entry, public goods, volunteer's, Battle of the Exes, Find the Unicorn): you implement them anyway, and their state is tiny, so the renderer is a few f-strings ("Last round the group total was 17 too high. Choose 0–50."). You write it, but it's trivial. 
Tier 3 — pixel/grid/vector observations ? needs a real text-abstraction layer (the hard case)
Same problem as Melting Pot: these emit pixels or feature vectors, so an LLM needs a Mosquera/Hypothetical-Minds-style grid?text renderer and a text?move parser.
    • sequential_social_dilemma_games (Cleanup/Harvest), RTG, SMAC. None ships an LLM text interface. (For StarCraft specifically, LLM-PySC2 is exactly someone building this — an observation wrapper that processes text observations per agent — global info, unit counts, positions/health, valid actions, last-step events, communication messages — and aggregates them into LLM query messages.)

Two frameworks to pull in so Claude Code doesn't rebuild this
    • TextArena (Guertler et al., 2025) — an open-source collection of more than fifty text-based games for training/evaluating agentic behaviour in LLMs, supporting single-, two-, and multi-player scenarios with an online platform and TrueSkill tracking. It already contains text-native versions of negotiation, secret-mafia/social-deduction, and bargaining games — likely covers several of your hidden-role/economic games out of the box. 
    • Game Reasoning Arena (OpenSpiel?LLM text) for the classic games. 
    • The Mosquera / Hypothetical Minds renderers for the grid envs (from the earlier discussion). 
Don't forget the action half
The language interface is two-directional, and the return path is where most LLM-game harnesses get fiddly. The standard pattern (seen across these works) is a rule-based parser with error feedback: a text interface that translates observations into natural-language descriptions and encodes agent chats into abstract action selections via keyword matching; if an agent produces an invalid action or nonsensical text, the interface returns a templated error message for correction. Build that once as a reusable ActionParser base with per-game validators, not per-game from scratch.
So your testbed's real spine is: TextRenderer + ActionParser per game, with the model behind Policy.act() — and the env (PettingZoo, Melting Pot, OpenSpiel, a repo, or your native code) sits underneath. Tier 1 gives you both halves; Tier 2 gives you one or makes both trivial; Tier 3 is where the engineering is.

That's the piece that turns this from an RL testbed into the LLM testbed you actually want.
